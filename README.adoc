= Capsium Viewer (web extension)
:toc: preamble

A browser extension (Chrome MV3; Firefox target available) that loads a
`.cap` Capsium package and renders it in the browser ‚ÄĒ a user-side Capsium
reactor.

== What it does

. The popup accepts a `.cap` (ZIP) package.
. The package is validated and normalized against the canonical Capsium
  schemas (ARCHITECTURE.md ¬ß2‚Äď6 in the monorepo root):
+
--
** `metadata.json` (required), `manifest.json`, `routes.json`, `storage.json`
   are parsed with zod; the legacy gem formats
   (`{"content": [{file, mime}]}`, `{"routes": [{path, target: {file}}]}`,
   `{"datasets": [...]}`, array dependencies) are accepted and normalized.
** `manifest.json` / `routes.json` are auto-generated when absent
   (¬ß3/¬ß4 rules: `content/` scan, `index`, HTML dual routes,
   dataset routes under `/api/v1/data/<id>`).
** When `security.json` is present, every file's SHA-256 checksum is
   verified (`crypto.subtle`); the package is **rejected** on mismatch.
** When `security.json` declares `digitalSignatures` (¬ß6a), the RSA-SHA256
   signature (`signature.sig`) is verified over the canonical signed payload
   ‚ÄĒ the bytes of every checksummed file, concatenated in sorted key order ‚ÄĒ
   using the packaged SPKI public key (`crypto.subtle`,
   RSASSA-PKCS1-v1_5 + SHA-256). Verification happens **before install** and
   the package is **rejected** with a clear error on mismatch; the popup
   surfaces it. (X.509 certificates are not importable by WebCrypto; ship an
   SPKI `PUBLIC KEY` PEM.)
** Encrypted packages (¬ß6b: `metadata.json` + `signature.json` envelope +
   `package.enc`) are detected on open. The popup then shows a private-key
   field; with the recipient's PKCS#8 PEM the DEK is unwrapped
   (RSA-OAEP SHA-256) and the inner zip decrypted (AES-256-GCM) via
   `crypto.subtle`, after which the package loads like any other. A wrong
   or absent key yields a clear, retryable error.
** Layered storage (¬ß5a): when `storage.json` declares `layers`
   (bottom ‚Üí top), every layer file is stored under its raw path and each
   request resolves the merged view **top ‚Üí bottom** ‚ÄĒ first hit wins.
   `.capsium-tombstones` deletions resolve 404 (the router renders one),
   `private` layers are hidden from dependent packages, and the manifest /
   routes auto-generate from the merged view.
** Composite packages (¬ß4a): the popup lists a package's declared
   `dependencies` and accepts the dependency `.cap` files in the same
   session (the guid must match a declared dependency). An installed
   dependency's `exported` routes/resources serve as a lower read-only
   layer under the dependent's URL space (its own routes win; `private`
   routes/resources and `private` layers are never served). Route
   inheritance attributes are honored: `remap` (matched path),
   `responseRewrite.body` (served as a body override), and
   `responseRewrite.headers` / `responseHeaders` (applied where blob:
   documents allow ‚ÄĒ see *Serving architecture*); `requestHeaders` were
   inert in the data:-URI core and are no longer emitted as rules.
** Basic auth (¬ß4b): when `authentication.json` enables `basicAuth`, every
   package URL resolves `locked` (the router renders the 401 body
   "authentication required") until the popup's credential form verifies
   against the packaged htpasswd file (bcrypt and apr1-MD5 hashes
   supported, same verifier as capsium-js). Credentials are asked **once
   per session**; afterwards the serving layer attaches
   `Authorization: Basic ‚Ä¶` to every request under the package's origin
   and the tab reopens. OAuth2 is parsed (config model only) ‚ÄĒ the
   extension is not a confidential client, so the authorization-code flow
   is documented as deferred.
--
. Package files are extracted **binary-safe** (raw bytes) and stored in the
  binary file store ‚ÄĒ OPFS (Origin Private File System) primarily, the
  Cache API as fallback. `chrome.storage.local` keeps only the package
  index and per-package metadata (no more base64 payloads, no more
  `unlimitedStorage`).
. HTML files get their relative URLs rewritten against their own in-package
  directory, pointing at the package's synthetic origin
  `https://<capId>.cap`.
. ONE `declarativeNetRequest` session rule per package redirects every
  `https://<capId>.cap<path>` URL to the extension's router page (the path
  travels in the URL fragment). Rule IDs are namespaced per package and
  removed when the package expires (30 min).
. A new tab opens at `https://<capId>.cap/`, which lands on the router
  page. The popup shows package info plus a mini content-validity view
  (¬ß7).

== Serving architecture

The data:-URI core (base64 in `chrome.storage.local` + one DNR rule per
route redirecting to a `data:` URI) is replaced by two pieces:

[source]
----
https://<capId>.cap<path>
        ‚Ēā  (1) ONE DNR session rule per package (regex substitution:
        ‚Ēā      path -> URL fragment)
        ‚Ėľ
chrome-extension://<id>/router.html#/serve/<capId>/<path>
        ‚Ēā  (2) resolve message -> background worker:
        ‚Ēā      route match, ¬ß5a layers/tombstones, ¬ß4a dependency refs,
        ‚Ēā      ¬ß4b basic-auth gate  =>  {store, fileCapId, path, MIME,
        ‚Ēā                              bodyOverride?, responseHeaders?}
        ‚Ēā  (3) bytes read from the file store (OPFS / Cache API)
        ‚Ėľ
sandbox.html  (opaque origin, no extension APIs)
        ‚Ēā  (4) HTML: subresource refs -> blob: URLs minted from stored
        ‚Ēā      bytes ("fetch proxy"), exact MIME via Blob type, then
        ‚Ēā      document.written into the sandbox page
        ‚Ēā      non-HTML: navigated to a typed blob: URL
        ‚Ėľ
packaged document (scripts run; fetch()/navigations to the .cap origin
are bridged back through the router by an injected shim)
----

Storage:: **OPFS is the primary store** (chosen deliberately over the Cache
API): it persists raw bytes without request/response or serialization
overhead, handles large package trees cheaply and drops a whole package
with one recursive `removeEntry`. The async `FileSystemFileHandle` API is
used (sync access handles exist only in dedicated workers, never in a
service worker). **Fallback:** when OPFS is missing or the probe write
fails (older engines, quota pressure), the Cache API
(`caches.open('capsium')`, Response bodies keyed by synthetic
never-fetched URLs) is used instead; the resolution tells the router
which backend holds the bytes. Content types are not stored in either
backend ‚ÄĒ they live in the `chrome.storage.local` metadata
(`path -> contentType`) and travel with resolutions.
Serving:: ONE DNR redirect rule per package (down from one per route) ‚ÄĒ
the ~5k-rule budget pressure, per-route rule bookkeeping and the base64
URL-size ceiling are gone. Route matching moved to serve time
(`lib/resolver.ts`), which makes dependency installs (¬ß4a) and basic-auth
unlocks (¬ß4b) take effect without any rule rebuild.
Permissions:: `unlimitedStorage` was **removed** (index + metadata fit the
default `chrome.storage.local` quota easily; bytes live in OPFS/Cache).
`router.html` is web-accessible to `https://*.cap/*` (required for DNR
redirects to extension pages); the Chrome build declares `sandbox.html`
as a sandbox page.
Response headers:: the Blob type now carries the exact `Content-Type` of
every served file (the data:-URI core embedded it in the URI; both beat
DNR, which cannot set response headers on redirects). A ¬ß4a
`Content-Security-Policy` response header is applied via a `<meta>` tag;
other custom response headers cannot be attached to blob: documents and
are documented as dropped (they were dropped by the data:-URI core too ‚ÄĒ
DNR cannot apply response headers to data: responses). The ¬ß4b
`Authorization: Basic` request-header rule is still emitted
package-wide after authentication.

== Development

Requires Node.js >= 22.

[source,sh]
----
npm ci              # install (also runs `wxt prepare`)
npm run dev         # dev server with hot reload (Chrome)
npm run dev:firefox # dev server (Firefox)
npm run test        # vitest suite (chrome.* APIs mocked via DI fakes)
npm run lint        # eslint (flat config)
npm run typecheck   # tsc --noEmit
npm run build       # production build: .output/chrome-mv3 + .output/firefox-mv2
npm run zip         # distributable zips in .output/
----

=== Load unpacked

. `npm run build`
. Chrome: `chrome://extensions` ‚Üí enable Developer mode ‚Üí *Load unpacked* ‚Üí
  select `.output/chrome-mv3/`.
. Click the extension icon, choose a `.cap` file (see `tests/` for fixtures).

== Architecture

`src/lib/` is pure and unit-tested separately from the browser glue:

* `model/` ‚ÄĒ zod schemas (single source of truth) for metadata / manifest /
  routes / storage / security / encryption / authentication, legacy
  normalization, auto-generation. Should later converge on `@capsium/core`
  (capsium-js) once published.
* `package-loader.ts` ‚ÄĒ .cap ‚Üí validated, normalized in-memory package.
* `signatures.ts` / `encryption.ts` ‚ÄĒ ¬ß6a signature and ¬ß6b decryption
  gates (WebCrypto).
* `layers.ts` ‚ÄĒ ¬ß5a overlay resolution (top ‚Üí bottom, tombstones,
  exported/private viewpoints).
* `composite.ts` / `resolver.ts` ‚ÄĒ ¬ß4a dependency refs and serve-time
  route resolution (own routes above inherited dependency routes; layered
  tombstones; body/header overrides).
* `auth/` ‚ÄĒ ¬ß4b htpasswd verification (bcrypt via bcryptjs, apr1 via the
  bundled RFC 1321 MD5, ported from capsium-js swsws).
* `store.ts` ‚ÄĒ transactional persistence: bytes to the file store,
  index + metadata in `chrome.storage.local` (rollback on partial
  failure).
* `file-store.ts` ‚ÄĒ the binary file store: OPFS backend (primary), Cache
  API backend (fallback), probe-based selection.
* `dnr.ts` ‚ÄĒ per-package namespaced DNR rule lifecycle (ONE router
  redirect rule + the ¬ß4b Authorization rule).
* `router-page.ts` / `sandbox-page.ts` ‚ÄĒ the serving pages: resolution
  orchestration, blob: fetch-proxy DOM pass, blob URL lifecycle, the
  injected fetch/navigation shim.
* `offscreen-rewriter.ts` ‚ÄĒ single persistent offscreen document with
  request correlation (Chrome); direct `DOMParser` in Firefox event pages.
* `background-service.ts` ‚ÄĒ orchestration (load ‚Üí rewrite ‚Üí store ‚Üí rule ‚Üí
  open tab; serve-time resolution for the router; expiry sweep;
  session-rule recovery after browser restart).
* `popup-view.ts` ‚ÄĒ DOM-building rendering (no `innerHTML` with package
  data).

`src/entrypoints/` holds the thin WXT entrypoints (background service
worker, popup, offscreen document, router page, sandbox page) that wire
the DI ports to `webextension-polyfill` / `chrome.*`.

== Known limitations / future work

* Serving is via the router + sandbox pages (see *Serving architecture*).
  The data:-URI core is gone; the trade-offs of the new core:
** blob: documents cannot carry arbitrary response headers ‚ÄĒ ¬ß4a
   `responseHeaders` / `responseRewrite.headers` are applied only for
   `Content-Security-Policy` (via `<meta>`); other response headers are
   dropped (they were inert in the data:-URI core too). `requestHeaders`
   companions are no longer emitted (they never had an observable effect).
** Packaged `fetch()` to the package origin is bridged through the router
   (dataset routes work from JS); packaged `XMLHttpRequest` to package
   URLs is NOT bridged and returns the router page ‚ÄĒ use `fetch()`.
** In-package link clicks and `window.open()` are relayed through the
   injected shim (a blob:-origin frame cannot follow the DNR redirect to
   an extension page). `<iframe src>` pointing at a package URL does not
   load (the data:-URI core couldn't frame data: documents either).
** ES modules with relative imports do not resolve from blob:-origin
   documents (same class of limitation as data: documents); classic
   scripts work. Inline `style` attributes / CSS `url()` references keep
   their pre-existing behavior (relative references were unresolvable in
   the data:-URI core too).
** CSP nuances: on Chrome the sandbox page's CSP allows `'unsafe-inline'`
   + `blob:` scripts (sandbox pages may relax it; extension pages may
   not). Firefox has no manifest sandbox pages (the iframe `sandbox`
   attribute provides the opaque origin) and its MV2 CSP validation
   forbids `'unsafe-inline'` ‚ÄĒ packaged EXTERNAL (blob:) scripts run on
   Firefox, packaged INLINE scripts are blocked there. Scripts from
   external (https:) origins are blocked by the sandbox CSP on both
   browsers (the data:-URI core ran them; blocking is a deliberate
   hardening, not a platform constraint).
** Firefox OPFS notes: the async OPFS API (`navigator.storage.getDirectory()`)
   works in Firefox ‚Č• 111 window/worker contexts, which covers the MV2
   event page used here; if it is missing or fails, the Cache API
   fallback engages automatically (probe at service-worker startup).
** Unknown package URLs now render the router's styled 404 instead of the
   browser's generic unreachable-host page (a deliberate improvement).
* Dependency version ranges are displayed but not semver-matched (the user
  picks the exact .cap files; guid identity is enforced). Transitive
  `capsium://` references inside a dependency's routes are not served.
* OAuth2 (¬ß4b) is config-model only: the extension is not a confidential
  client (client secrets could not be kept secret), so the
  authorization-code flow is deferred; browser-side PKCE is a possible
  later phase. htpasswd supports bcrypt and apr1-MD5; plaintext/DES/
  SHA-crypt entries are rejected with a precise error. Basic-auth
  credentials live in service-worker memory only (one prompt per session;
  the 401 body returns after a browser restart).
* Dynamic handler routes (`{path, method, handler}`) are parsed but not
  executed (reactors respond 501; execution deferred).
* Writes to the topmost writable layer are not supported (the viewer is
  read-only; ¬ß5a write semantics deferred).
* Firefox target builds (MV2 event pages) but is best-effort; Chrome is the
  primary target.
