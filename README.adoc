= Capsium Viewer (web extension)
:toc: preamble

A browser extension (Chrome MV3; Firefox target available) that loads a
`.cap` Capsium package and renders it in the browser — a user-side Capsium
reactor.

== What it does

. The popup accepts a `.cap` (ZIP) package.
. The package is validated and normalized against the canonical Capsium
  schemas (ARCHITECTURE.md §2–6 in the monorepo root):
+
--
** `metadata.json` (required), `manifest.json`, `routes.json`, `storage.json`
   are parsed with zod; the legacy gem formats
   (`{"content": [{file, mime}]}`, `{"routes": [{path, target: {file}}]}`,
   `{"datasets": [...]}`, array dependencies) are accepted and normalized.
** `manifest.json` / `routes.json` are auto-generated when absent
   (§3/§4 rules: `content/` scan, `index`, HTML dual routes,
   dataset routes under `/api/v1/data/<id>`).
** When `security.json` is present, every file's SHA-256 checksum is
   verified (`crypto.subtle`); the package is **rejected** on mismatch.
** When `security.json` declares `digitalSignatures` (§6a), the RSA-SHA256
   signature (`signature.sig`) is verified over the canonical signed payload
   — the bytes of every checksummed file, concatenated in sorted key order —
   using the packaged SPKI public key (`crypto.subtle`,
   RSASSA-PKCS1-v1_5 + SHA-256). Verification happens **before install** and
   the package is **rejected** with a clear error on mismatch; the popup
   surfaces it. (X.509 certificates are not importable by WebCrypto; ship an
   SPKI `PUBLIC KEY` PEM.)
** Encrypted packages (§6b: `metadata.json` + `signature.json` envelope +
   `package.enc`) are detected on open. The popup then shows a private-key
   field; with the recipient's PKCS#8 PEM the DEK is unwrapped
   (RSA-OAEP SHA-256) and the inner zip decrypted (AES-256-GCM) via
   `crypto.subtle`, after which the package loads like any other. A wrong
   or absent key yields a clear, retryable error.
** Layered storage (§5a): when `storage.json` declares `layers`
   (bottom → top), every layer file is stored under its raw path and each
   request resolves the merged view **top → bottom** — first hit wins.
   `.capsium-tombstones` deletions resolve 404 (no DNR rule is installed),
   `private` layers are hidden from dependent packages, and the manifest /
   routes auto-generate from the merged view.
** Composite packages (§4a): the popup lists a package's declared
   `dependencies` and accepts the dependency `.cap` files in the same
   session (the guid must match a declared dependency). An installed
   dependency's `exported` routes/resources serve as a lower read-only
   layer under the dependent's URL space (its own routes win via a higher
   DNR priority; `private` routes/resources and `private` layers are never
   served). Route inheritance attributes are honored: `remap` (matched
   path), `responseRewrite.body` (baked into the served data: URI), and
   `responseRewrite.headers` / `responseHeaders` / `requestHeaders`
   (companion `modifyHeaders` rules — see the header caveat below).
--
. Package files are extracted **binary-safe** (raw bytes; uniform base64 in
   `chrome.storage.local` — PDFs/images are no longer corrupted).
. HTML files get their relative URLs rewritten against their own in-package
   directory, pointing at the package's synthetic origin
   `https://<capId>.cap`.
. One `declarativeNetRequest` session rule per route redirects
   `https://<capId>.cap<path>` to the file's data: URI. Rule IDs are
   namespaced per package and removed when the package expires (30 min).
. A new tab opens at `https://<capId>.cap/`. The popup shows package info
  plus a mini content-validity view (§7).

== Development

Requires Node.js >= 22.

[source,sh]
----
npm ci              # install (also runs `wxt prepare`)
npm run dev         # dev server with hot reload (Chrome)
npm run dev:firefox # dev server (Firefox)
npm run test        # vitest suite (chrome.* APIs mocked via DI fakes)
npm run lint        # eslint (flat config)
npm run typecheck   # tsc --noEmit
npm run build       # production build: .output/chrome-mv3 + .output/firefox-mv2
npm run zip         # distributable zips in .output/
----

=== Load unpacked

. `npm run build`
. Chrome: `chrome://extensions` → enable Developer mode → *Load unpacked* →
  select `.output/chrome-mv3/`.
. Click the extension icon, choose a `.cap` file (see `tests/` for fixtures).

== Architecture

`src/lib/` is pure and unit-tested separately from the browser glue:

* `model/` — zod schemas (single source of truth) for metadata / manifest /
  routes / storage / security, legacy normalization, auto-generation.
  Should later converge on `@capsium/core` (capsium-js) once published.
* `package-loader.ts` — .cap → validated, normalized in-memory package.
* `signatures.ts` / `encryption.ts` — §6a signature and §6b decryption
  gates (WebCrypto).
* `layers.ts` — §5a overlay resolution (top → bottom, tombstones,
  exported/private viewpoints).
* `composite.ts` / `serving.ts` — §4a dependency refs and composite rule
  specs (own routes above inherited dependency routes).
* `store.ts` — transactional `chrome.storage.local` persistence (rollback on
  partial failure).
* `dnr.ts` — per-package namespaced DNR rule lifecycle.
* `offscreen-rewriter.ts` — single persistent offscreen document with
  request correlation (Chrome); direct `DOMParser` in Firefox event pages.
* `background-service.ts` — orchestration (load → rewrite → store → rules →
  open tab; expiry sweep; session-rule recovery after browser restart).
* `popup-view.ts` — DOM-building rendering (no `innerHTML` with package
  data).

`src/entrypoints/` holds the thin WXT entrypoints (background service
worker, popup, offscreen document) that wire the DI ports to
`webextension-polyfill` / `chrome.*`.

== Known limitations / future work

* Serving is via DNR redirects to base64 data: URIs (kept from the original
  design). Very large resources inflate by ~33% and depend on URL-length
  limits. An OPFS / Cache-API-based serving layer (replacing data: URIs) is
  explicitly deferred. Tombstoned/absent package URLs have no rule, so the
  browser shows its generic unreachable-host page rather than a styled 404.
  Header rules (`headers`, `responseHeaders`, `responseRewrite.headers`)
  are emitted as companion `modifyHeaders` rules, but DNR cannot apply
  response headers to `data:` responses — they take effect only once
  serving moves to OPFS/Cache API.
* Dependency version ranges are displayed but not semver-matched (the user
  picks the exact .cap files; guid identity is enforced). Transitive
  `capsium://` references inside a dependency's routes are not served.
* Dynamic handler routes (`{path, method, handler}`) are parsed but not
  executed (reactors respond 501; execution deferred).
* Writes to the topmost writable layer are not supported (the viewer is
  read-only; §5a write semantics deferred).
* Firefox target builds (MV2 event pages) but is best-effort; Chrome is the
  primary target.
* `unlimitedStorage` is kept because packages are stored base64-encoded in
  `chrome.storage.local` (10 MB default quota is too small).
